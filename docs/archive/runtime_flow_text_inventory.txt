P	163	The target folder is only suggested here. It is not created yet.

P	179	When the user clicks `Convert Project`, `btnConvertClick` runs.
P	181	This is the handoff point from the user-facing UI into the real conversion pipeline.
P	183	The convert handler performs a strict set of validations before it starts any worker thread.

P	187	If `FConversionThread` is already assigned, it shows:
P	189	`A conversion is already running.`
P	191	and exits immediately.
P	193	6.2. It reads and trims the source and target edits
P	195	It stores:
P	197	`Src := edtSource.Text.Trim`
P	198	`Dst := edtTarget.Text.Trim`
P	200	6.3. It validates the source and target paths
P	202	It stops and shows a message if:
P	204	source folder is blank
P	205	target folder is blank
P	206	source folder does not exist
P	207	source folder and target folder resolve to the same actual path
P	209	If source and target are the same, it shows:
P	211	`Please choose a different output folder. The source and target folders should not be the same.`
P	213	It also switches the tab back to `tabProjectScan` when path validation fails, so the operator returns to the folder-selection screen.
P	215	6.4. It resets the visible run state
P	217	Once validation passes, it:
P	219	clears the log
P	220	adds `Preparing conversion...`
P	221	switches the active tab to `tabIssues`

P	225	6.5. It captures the current conversion options
P	227	It reads:

P	230	recursive on/off
P	231	critical-areas on/off
P	232	data-aware on/off
P	233	third-party on/off
P	234	WinAPI on/off
P	235	mapping packs enabled by default, with the folder resolved from the executable-side mapping_packs directory unless overridden in the conversion context
P	237	6.6. It primes the run-summary fields

P	241	last output path becomes the chosen target path
P	242	last report paths are cleared
P	243	pending files processed = 0
P	244	pending files converted = 0
P	245	pending files with errors = 0

P	247	pending manual review count = 0

P	276	6.8. It creates the worker thread
P	278	The actual conversion work does not happen on the UI thread.
P	280	`btnConvertClick` creates an anonymous thread and starts it. That thread calls:
P	282	`ConversionThreadProc(Src, Dst, FileTypeIndex, Recursive, EnableCritical, EnableDataAware, EnableThirdParty, EnableWinAPI)`
P	284	Inside the thread wrapper:
P	286	`StatusText` starts as `Conversion failed`

P	291	it then queues `CompleteConversionOnUIThread` back to the main thread using `TThread.Queue`

P	417	10. Mapping-Pack Load Rules
P	418	Mapping-pack action matrix
P	419	Before conversion work relies on mappings, `TComponentMapper.LoadMappings` loads the built-in mapper database and then calls `LoadMappingPacks`.
P	420	Mapping packs are optional external JSON files. In v4.1.6 they live in `mapping_packs` and are transparent in the text and HTML reports.
P	421	Each pack rule must use a valid action: `convert`, `partial`, `manual_review`, `detect_only`, or `preserve`.
P	422	Malformed packs are reported instead of crashing the conversion. Duplicate rules are handled deterministically by sorted file loading and normal mapper precedence.
P	423	During DFM generation, mapping-pack rules can convert, partially convert, preserve, detect only, or force manual review. Each use is recorded in `FContext.MappingPackUsages` for report output.
P	424	Low-confidence rules are not silently converted; they are reported for manual review.
P	426	`FFileManager.Reset`
P	427	`FFileManager.PrepareOutput`
P	429	These are not the same thing.
P	431	`Reset` scans the source side and builds the list of candidate files.
P	433	`PrepareOutput` prepares the target side.
P	435	11. Source Scan Rules: What Files Are Collected
P	437	The file manager starts by clearing its internal file list and resetting its index.
P	439	It then verifies that the source directory exists. If it does not, it adds an error issue and stops.
P	441	11.1. The output folder is excluded from scanning
P	443	If the output folder sits under the source tree, the file manager deliberately skips it so the converter does not re-read its own generated output as input.
P	445	11.2. Excluded folders
P	447	The file manager excludes any path segment matching:
P	449	`backup`
P	450	`backup copy`
P	451	`win32`
P	452	`win64`
P	453	`deploy`
P	454	`__history`
P	455	`.git`
P	456	`.svn`
P	457	`recovery_generic`
P	458	`debug`
P	459	`release`
P	461	11.3. Recursive vs top-only scan
P	463	If recursive mode is on, subdirectories are scanned.
P	465	If recursive mode is off, only the top directory is scanned.
P	467	11.4. File-type filtering
P	469	After a file is found, it must match the currently selected file type:
P	471	`.pas` only
P	472	`.dfm` only
P	473	both `.pas` and `.dfm`
P	475	11.5. File order
P	477	After collection, the file list is sorted.
P	479	That means the converter processes files in sorted order, not raw filesystem discovery order.
P	481	12. Output Preparation Rules
P	483	`PrepareOutput` is where the converter actually makes the output directory real if it does not exist already.
P	485	This is the point where the target folder can be created.
P	487	It does the following:
P	489	validates that an output path exists in the options
P	490	if the output folder does not exist, calls `ForceDirectories`
P	491	removes stale build-artifact folders from the output path if they exist:
P	492	- `Win32`
P	493	- `Win64`
P	494	- `Debug`
P	495	- `Release`
P	496	- `__history`
P	498	So the output path is not just created. It is also cleaned of the most common stale build subfolders before file conversion begins.
P	500	13. File-Processing Loop
P	502	After scan and output preparation, the engine asks the file manager for the file count and logs:
P	504	`Found X files to process`
P	506	It then loops while:
P	508	more files exist
P	509	the run has not been cancelled
P	511	For each file it:
P	513	gets the next full file path
P	514	increments the processed counter
P	515	logs `[n/total] Processing: filename`
P	516	routes the file by extension:
P	517	- `.pas` -> `ProcessPasFile`
P	518	- `.dfm` -> `ProcessDfmFile`
P	520	If an exception occurs around a file-level pass, the engine:
P	522	increments file-error count
P	523	logs a failed message
P	524	adds a blocking issue to the context

P	571	15. DFM File Flow
P	572	DFM pipeline summary
P	574	When a DFM file is processed, this is the actual sequence.
P	576	15.1. The DFM is loaded
P	578	The engine logs that it is processing the DFM and sets `FContext.CurrentFile`.
P	580	Inside the orchestrator, the DFM parser loads the file through `LoadDFM`.
P	582	If the DFM is binary:
P	584	it detects binary DFM streams using Delphi stream-format detection, including modern resource-header forms and raw `TPF0` streams
P	585	converts binary DFM to text with the parser-side binary conversion path, trying the resource and binary stream conversion routes before failing explicitly
P	587	If the DFM is already text:
P	589	it uses the shared safe text-read path and tries UTF-8
P	590	falls back to ANSI when needed
P	592	15.2. The DFM text is parsed into a component tree
P	594	The parser:
P	596	clears the old component list
P	597	resets the current component and collection state
P	598	walks the DFM line by line
P	599	parses `object` / `inherited` declarations
P	600	parses collections and subcomponents
P	601	parses properties and events
P	602	records original line numbers for properties and events
P	604	The result is an internal tree of `TDFMComponent` objects describing the source form.
P	606	15.3. Component analysis happens before FMX output is generated
P	608	The orchestrator then:
P	610	recursively analyzes components for third-party detection if third-party handling is enabled
P	611	runs data-aware analysis if data-aware handling is enabled
P	612	clears old data-binding metadata if data-aware handling is disabled
P	614	15.4. The FMX form text is generated
P	616	The parser then converts the parsed DFM tree into FMX text by calling:
P	618	`FDfmParser.Convert(FComponentMapper)`
P	620	The FMX generator:
P	622	takes the parsed root component
P	623	generates the FMX object tree
P	624	uses the component mapper to translate VCL classes, properties, and events
P	625	records third-party and mapping-pack usage for the final report
P	626	honors mapping-pack actions such as convert, partial, manual_review, detect_only, and preserve
P	627	normalizes generated FMX values such as align constants
P	628	emits FMX text for the output form
P	630	After that, the orchestrator applies the full-text memo promotion pass to the generated FMX text and replaces the original DFM text with the generated FMX version.
P	632	It then logs:
P	634	`DFM conversion successful: <file>`
P	636	15.5. The output filename is changed from `.dfm` to `.fmx`
P	638	When the file manager saves a converted DFM, it automatically changes the output extension from `.dfm` to `.fmx`.
P	640	That means:
P	642	`SomeForm.dfm`
P	644	becomes:
P	646	`SomeForm.fmx`
P	648	in the FMX output tree.

P	719	`.dpr`

P	724	It only audits files touched during the current run. That is done by comparing file write times against the run start time. This is specifically meant to avoid stale leftovers in reused output folders from polluting the report for the current conversion.

P	789	20. Engine Completion Logic
P	791	After the report is written, the engine decides how to log completion.

P	801	it returns `True`
P	803	If neither blocking nor manual-review issues exist:
P	805	it logs that conversion completed cleanly
P	806	it returns `True`
P	808	Important detail:
P	810	The worker-thread status string shown back in the main form is decided separately in `ConversionThreadProc`, but it uses the same basic logic.
P	812	21. How the UI Is Restored After the Worker Thread Finishes
P	814	When the background thread ends, it does not directly update the UI.
P	816	Instead it queues `CompleteConversionOnUIThread`.
P	818	That method runs back on the main thread and does exactly three things:
P	820	frees the conversion thread object
P	821	calls `UpdateUIForConversion(False)` to re-enable controls
P	822	calls `UpdateArtifactsAfterConversion(TargetPath, StatusText)`
P	824	22. What `UpdateArtifactsAfterConversion` Does
P	826	This is the method that makes the finished run visible to the operator.
P	828	It:
P	830	marks that a run has completed
P	831	copies all pending counters into the last-run fields
P	832	stores the last output path
P	833	computes the expected text and HTML report paths
P	834	picks the preferred report path, preferring HTML over text
P	835	updates the status labels
P	836	updates dashboard output and report labels
P	837	updates issues-tab output and report labels
P	838	rebuilds the summary sentence
P	839	re-enables open-report, print-report, and open-output actions as appropriate
P	841	So this is the moment when the UI leaves the "running" state and becomes a post-run review screen.
P	843	23. What the User Can Do After the Run
P	845	Once conversion is complete and the UI is unlocked, the operator can:
P	847	open the output folder
P	848	open the preferred report
P	849	print the preferred report
P	850	reset the converter back to its startup state
P	852	23.1. Open Output
P	854	`btnOpenOutputClick`:
P	856	checks whether `FLastOutputPath` exists
P	857	if not, shows `There is no converted output folder to open yet.`
P	858	otherwise uses Windows `ShellExecute('open', path)` to open the folder
P	860	23.2. Open Report
P	862	`btnOpenReportClick`:
P	864	asks for the preferred report path
P	865	prefers HTML if it exists
P	866	otherwise uses the text report if it exists
P	867	if neither exists, shows `There is no generated report to open yet.`
P	868	otherwise opens the report with `ShellExecute('open', path)`
P	870	23.3. Print Report
P	872	`btnPrintReportClick`:
P	874	resolves the preferred report path
P	875	if none exists, shows `There is no generated report to print yet.`
P	876	attempts `ShellExecute('print', reportPath, hideWindow=True)`
P	877	if direct print is not available, it opens the report instead and tells the user that direct printing was not available
P	879	24. Reset Flow
P	881	If the user clicks `Reset`, `btnResetClick` runs.
P	883	If a conversion thread is still active, reset is blocked with:
P	885	`Please wait for the current conversion to finish before resetting.`
P	887	If no conversion is running, `ResetToStartupState` is called.
P	889	That method:
P	891	clears source and target edits
P	892	restores file type to `Both PAS and DFM`
P	893	re-enables all main option checkboxes to their default checked state
P	894	clears pending completion and message fields
P	895	restores the startup log text
P	896	rebuilds and reloads the reference-page content
P	897	resets artifacts and run summary
P	898	updates the rule summary
P	899	marks the UI as not converting
P	900	scrolls the tab scroll boxes back to the top
P	901	returns the active tab to `tabProjectScan`
P	902	applies responsive layout
P	903	moves focus back to the source edit if possible
P	905	So reset is a full UI-state reset, not just a log clear.
P	907	25. Close Protection While a Run Is Active
P	909	If the user tries to close the program while a conversion is still running, `FormCloseQuery` blocks the close.
P	911	It only allows the program to close when `FConversionThread` is not assigned.
P	913	If a run is active, it shows:
P	915	`A conversion is still running. Please wait for it to finish before closing the program.`
P	917	That means the converter deliberately prevents shutdown in the middle of an active conversion pass.
P	919	26. Thread Boundary Summary
P	921	There are two major runtime domains in the program:
P	923	26.1. UI thread
P	925	The UI thread handles:
P	927	form startup
P	928	folder selection
P	929	validation
P	930	starting the worker thread
P	931	log display updates
P	932	synchronized error dialogs
P	933	final completion UI updates
P	934	post-run actions like open-report, print-report, open-output, and reset
P	936	26.2. Worker thread
P	938	The worker thread handles:
P	940	conversion context creation
P	941	engine construction
P	942	source scan
P	943	output preparation
P	944	per-file Pascal conversion
P	945	per-file DFM conversion
P	946	project generation
P	947	generated-output audit
P	948	report generation
P	949	final run-status determination
P	951	The worker thread never directly finalizes the UI. It hands completion back through `TThread.Queue`.
P	953	27. Exact Start-to-Finish Narrative in Plain English
P	955	If you want the whole thing in one straight-through human narrative, this is it:
P	957	The program starts.
P	958	`FormCreate` runs.
P	959	The form wires its runtime events, loads the file-type options, defaults all major conversion modules to on, loads the mapping-reference material, resets all run artifacts, and sets the active tab to Project Scan.
P	960	`FormShow` runs.
P	961	The window maximizes, the theme/layout pass runs, and the embedded reference browsers are prepared.
P	962	The form is displayed and waits for user activity.
P	963	The user chooses a source folder.
P	964	The converter writes that folder into `edtSource`.
P	965	If the target folder is still blank, the converter suggests a sibling folder named `<SourceName> - FMX Output`.
P	966	The user can accept that target or browse for another target folder.
P	967	Nothing is created yet just from selecting folders.
P	968	The user clicks `Convert Project`.
P	969	The converter validates source and target.
P	970	It rejects blank paths, missing source folders, or a source path equal to the target path.
P	971	If validation succeeds, it clears the log, switches to the Issues tab, resets pending counters, updates output/report labels, disables interactive controls, changes the button text to `Converting...`, and starts a worker thread.
P	972	The worker thread creates a conversion context and copies in all selected options, including mapping-pack settings.
P	973	The worker thread creates the engine.
P	974	The engine resets its counters and issues, loads built-in/user/pack mappings through the mapper path, and logs the start banner.
P	975	The file manager scans the source tree, skipping excluded directories and skipping the output tree if it sits under the source tree.
P	976	The output directory is created if needed, and stale build subfolders under the output folder are deleted.
P	977	The engine loops through the sorted list of eligible `.pas` and `.dfm` files.
P	978	Each Pascal file is read, parsed, transformed through the orchestratorâ€™s multi-step pipeline, and saved into the output tree in relative position.
P	979	Each DFM file is loaded through the parser, converted from binary to text if needed, parsed into components, translated into FMX text with built-in and mapping-pack rules, renamed to `.fmx`, and saved into the output tree in relative position.
P	980	After source files finish, the project generator creates transformed `.dpr`, `.dproj`, and `.deployproj` files and copies or stages project assets and runtime companion files.
P	981	The engine audits the generated output that was touched during the current run.
P	982	The engine writes both the text and HTML conversion reports into the output folder, including loaded mapping packs, mapping-pack usage, grouped manual-review items, and ordinary issues.
P	983	The engine decides whether the run was clean, manual-review, or blocking.
P	984	The worker thread stores the final status and queues the completion handler back to the main thread.
P	985	The UI thread receives that completion callback.
P	986	The UI re-enables all controls, stores the final output and report paths, updates the dashboard and issues panels, and enables the report/output actions.
P	987	The user can now open the output folder, open the HTML report, print the report, inspect the generated FMX project, or reset the converter for another run.
